# Repository Skill Format

> Reference for: create-skills
> Load when: Bootstrapping a repository, scaffolding a new skill, or reviewing catalog conventions

## Repository Layout

A compatible skill catalog repository requires:

```
catalog.json          ← root catalog (formatVersion, repo, ref, skills[])
skills/
  <skill-id>/
    SKILL.md          ← primary skill file (advanced format)
    skill.json        ← local manifest (generated by CLI on install)
    agents/
      openai.yaml     ← OpenAI Codex / ChatGPT agent config
    references/
      <topic>.md      ← domain knowledge loaded on demand
    scripts/
      <name>.js       ← executable Node.js scripts
```

A bootstrapped repository must include `skills/create-skills/` and register it in the root catalog.

## SKILL.md Advanced Format

Every skill must use the advanced format. All sections are required.

### Frontmatter

```yaml
---
name: "Human Readable Name"
description: "One-sentence summary. Activates when you say 'trigger phrase one', 'trigger phrase two', ..."
---
```

- `name` — title-cased human name.
- `description` — used in catalog and as the AI's activation hint. **Include trigger patterns** so the AI knows when to load the skill.

### Required Sections

| Section | Purpose |
|---------|---------|
| `# Name` + overview paragraph | Brief statement of when to use the skill |
| `## Core Workflow` | Numbered steps (3–7) that the AI must follow |
| `## Reference Guide` | Table: Topic / Reference file / When to load |
| `## Bundled Scripts` | Table: Script / Purpose / Usage command |
| `## Constraints` | MUST DO and MUST NOT DO rules |
| `## Output Template` | Example or schema of the expected output |
| `## References` | Authoritative external links |

### Core Workflow format

```markdown
## Core Workflow

1. **Step name** — What to do and why.
2. **Step name** — What to do and why.
...
```

Each step must be actionable. Avoid vague steps like "think about the task."

### Reference Guide format

```markdown
## Reference Guide

| Topic | Reference | When to load |
|-------|-----------|--------------|
| Error Patterns | `references/error-patterns.md` | When writing error handling logic |
| Language Specifics | `references/error-by-language.md` | When language-specific syntax is needed |
```

### Constraints format

```markdown
## Constraints

**MUST DO**
- Rule one.
- Rule two.

**MUST NOT DO**
- Anti-pattern one.
- Anti-pattern two.
```

## Reference Files (`references/*.md`)

Each reference file documents domain knowledge for one topic. The AI loads a reference file when the Reference Guide says to.

### Required structure

```markdown
# Topic Title

> Reference for: <skill-id>
> Load when: <condition>

## Overview

Short explanation of the domain.

## Key Concepts

Patterns, conventions, and rules with code examples.

## Examples

\`\`\`language
// Working code example
\`\`\`

## Anti-Patterns

What to avoid and why.
```

Guidelines:
- Use fenced code blocks with language identifiers.
- Include one real example per concept — avoid toy examples.
- Cross-reference other reference files with `references/<name>.md` paths.
- Keep each file focused on one topic; split if a topic grows beyond ~200 lines.

## Scripts (`scripts/*.js`)

Scripts are executable Node.js ESM files the AI can call directly.

### Conventions

- Add a shebang: `#!/usr/bin/env node`
- Document usage in the header comment block.
- Support `--help` to print usage and exit 0.
- Exit 0 on success, exit 1 on failure.
- Support `--json` for machine-readable output when the script produces structured results.
- Accept `--root <path>` when the script needs to operate on a repository.

### Scaffold a placeholder script

```javascript
#!/usr/bin/env node
/**
 * example.js
 * TODO: describe what this script does.
 *
 * Usage:
 *   node skills/<id>/scripts/example.js [--option value]
 *
 * Exit codes:
 *   0 — success
 *   1 — error
 */

import { parseArgs } from "node:util";

const { values } = parseArgs({
  options: {
    help:  { type: "boolean", short: "h" },
    input: { type: "string" },
  },
});

if (values.help) {
  console.log("Usage: example.js [--input <value>]");
  process.exit(0);
}

// TODO: implement
console.log("Done.");
```

## Catalog Registration (`catalog.json`)

```json
{
  "id": "my-skill",
  "name": "My Skill",
  "version": "0.1.0",
  "description": "One-line description matching SKILL.md frontmatter.",
  "path": "skills/my-skill",
  "entry": "SKILL.md",
  "files": [
    "SKILL.md",
    "agents/openai.yaml",
    "references/domain.md",
    "scripts/validate.js"
  ],
  "compatibility": ["codex", "copilot", "cline", "cursor", "claude", "gemini", "windsurf"],
  "tags": ["topic", "workflow"],
  "author": "your-handle"
}
```

Rules:
- `id` must match the folder name under `skills/`.
- `files[]` must list every file the CLI should download on install — including all `references/*.md` and `scripts/*.js`.
- Do **not** include `skill.json` in `files[]`.
- Skills must be sorted by `id` in `catalog.json`.
- `compatibility` should use canonical adapter ids: `codex`, `copilot`, `cline`, `cursor`, `claude`, `gemini`, `windsurf`.

## CLI Usage

### Scaffold a new skill (advanced format)

```bash
node skills/create-skills/scripts/init_repo_skill.js \
  --skill-id my-skill \
  --name "My Skill" \
  --description "Handles my domain workflow. Activates when you say 'do X'." \
  --references domain-concepts,best-practices,anti-patterns \
  --tags workflow,domain \
  --root .
```

The `--references` flag generates a placeholder `references/<topic>.md` file for each comma-separated name and adds it to `files[]` automatically.

### Bootstrap a new skill repository

```bash
node skills/create-skills/scripts/bootstrap_skill_repo.js \
  --root /path/to/new-repo \
  --repo myorg/my-skills
```

### Validate catalog consistency

```bash
node skills/create-skills/scripts/check_catalog.js --root .
```
